```
severity: high
file: signal_smt.py
line: 545-553
issue: New exit_invalidation_* codes from manage_position silently leak state in live trading
detail: manage_position() now returns "exit_invalidation_mss", "exit_invalidation_cisd", or
"exit_invalidation_smt" when the corresponding flags are enabled. In _process_managing(), the
exit-type dispatch at lines 545-553 only handles "exit_tp", "exit_stop", "exit_market", and
"exit_session_end". Any unrecognised code falls into the `elif result != "exit_session_end": return`
branch which bails silently without printing an exit line, without clearing _position, without
updating _last_exit_ts, and without resetting _state to "SCANNING". The position remains
permanently open for the rest of the session; no further exits are attempted. In production this
means the live module gets stuck in MANAGING state indefinitely until the process is restarted.
The flags all default to False so this is dormant at the current default config, but any optimizer
run or manual test that sets INVALIDATION_MSS_EXIT=True will trigger it on the first invalidation bar.
suggestion: Add an elif branch covering the three new codes before the final elif.
  For each, use float(bar.close) as exit_price (mark-to-market, consistent with session_close):
    elif result in ("exit_invalidation_mss", "exit_invalidation_cisd", "exit_invalidation_smt"):
        exit_price = float(bar.close)
  This brings live behaviour into parity with the backtest (_build_trade_record already uses
  bar["Close"] for all non-TP/stop exits).
```

issue: smt_type column in trades.tsv is always empty when generated by the backtest
detail: _write_trades_tsv includes "smt_type" in fieldnames (line 657). However,
_build_trade_record (lines 90-157) never writes "smt_type" to the trade dict it returns.
The position dict does contain smt_type (it is spread from the signal via {**signal, ...}),
but _build_trade_record does not read it. csv.DictWriter with extrasaction="ignore" silently
emits an empty string for any missing key, so every smt_type cell in trades.tsv will be "".
The test test_write_trades_tsv_includes_smt_type passes only because it builds a trade dict
manually with smt_type already present — it does not exercise the _build_trade_record path.
suggestion: Add smt_type to _build_trade_record's returned trade dict:
  "smt_type": position.get("smt_type", "wick"),
  Place it alongside the other quality/diagnostic fields (after line 150).

```
severity: low
file: strategy_smt.py
line: 573-576 (screen_session), backtest_smt.py line 352-361 (WAITING_FOR_ENTRY), line 406-415 (REENTRY_ELIGIBLE)
issue: TP selection uses truthy-or fallback instead of explicit None check
detail: The OVERNIGHT_RANGE_AS_TP TP selection uses:
  _tp = overnight_range.get("overnight_low") or day_tdo  (for shorts)
  _tp = overnight_range.get("overnight_high") or day_tdo (for longs)
The "or" operator falls back to day_tdo when overnight_low/overnight_high is falsy — which
includes 0.0 in addition to None. For MNQ futures (prices 15,000+), a 0.0 value is physically
impossible, so this is a benign concern in practice. However, semantically the intent is "if
value is None, use TDO" not "if value is falsy, use TDO". The same pattern appears in three
separate places (screen_session, WAITING_FOR_ENTRY, REENTRY_ELIGIBLE), creating mild DRY
concern and risk that lower-valued instruments would silently use TDO.
suggestion: Replace `x or fallback` with `x if x is not None else fallback` at all three sites.
  Alternatively, extract a helper _select_tp(direction, overnight, midnight_open, tdo) and
  call it in all three places to eliminate the repetition.
```

## Summary

The changeset is logically correct and well-structured. All 89 tests pass. Three issues were identified:

1. **High**: `signal_smt.py` does not handle the three new `exit_invalidation_*` exit codes — when any of these flags is enabled, live trading silently leaks state and the position never closes. This is dormant at the current defaults (all three flags default to `False`) but will trigger as soon as any flag is set to `True`.

2. **Medium**: `smt_type` is added to the TSV fieldnames but never written to the trade record dict, so the column will always be empty in produced `trades.tsv` files. The new test for this feature passes only because it bypasses `_build_trade_record`.

3. **Medium/Low**: Stale comment in `_build_trade_record` and truthy-or fallback pattern for TP selection (benign for MNQ but semantically imprecise, repeated in three places).
